Synchronous and Asynchronous Response Support

Details regarding synchronous and synchronous response support

Both synchronous and asynchronous responses utilize the same ActionResponse model, but display a different status.

Synchronous Responses

Synchronous repsonses are the preferred option in every case; always try to return synchronous responses when your system supports them. Synchronous repsonses must always have a ResponseStatus that is not 0 / AsyncResponsePending.

Example Synchronous ActionResponse
  • 1.{
  • 2.   "ResponseBody": {},
  • 3.   "Status": "Complete",
  • 4.   "PendingUri": null,
  • 5.   "Errors": []
  • 6. }

Asynchronous Responses

It is possible that your system may not be able to perform some operations synchronously. For example, creating hundreds of product listings may take a few minutes to process, longer to process than is reasonable to wait for an HTTP response. In these cases, your API will return a null ResponseBody in its ActionResponse and a fully-qualified URL for ChannelAdvisor to poll.

Example Asynchronous ActionResponse
  • 1.{
  • 2.   "ResponseBody": null,
  • 3.   "Status": "AsyncResponsePending",
  • 4.   "PendingUri": "http://api.your-url.com/v1/pending-responses?workItemID=123456",
  • 5.   "Errors": []
  • 6. }

ChannelAdvisor will poll that URL every few seconds until it returns a valid response with a finalized ResponseStatus (not 0 / AsyncResponsePending).

Example Finalized Asynchronous ActionResponse
  • 1.{
  • 2.   "ResponseBody": {},
  • 3.   "Status": "Complete",
  • 4.   "PendingUri": null,
  • 5.   "Errors": []
  • 6. }

Asynchronous Polling Timeout

Because the ChannelAdvisor integration currently utilizes a synchronous feed process that waits for asynchronous responses to complete, no other work can continue for a feed while the feed waits for the polling URL to return a final status. For this reason, all asynchronous repsonses must complete within the configured asynchronous response timeout window (5 minutes by default). If your API is unable to respond within the allowed time window, the integration will give up, error the request internally, and move on to other work. This timeout will be treated as a system error. Currently, the only exception to this rule is the Create and update products endpoint when the Long-running batch workflow is being used.

If you find that a 5 minute timeout is insufficient for your system, please email your technical contact.
